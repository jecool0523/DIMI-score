# DIMI-scroe / README.md

> **"학교 체육대회의 모든 점수와 공지를 실시간으로, 끊김 없이 전달합니다."**
> 

DIMI-score는 제25회 체육대회 운영을 위해 개발된 실시간 점수판 및 관리 시스템입니다. 체육대회 당일, 야외 운동장이나 체육관에 있는 학생들에게 종목별 점수, 진행 상황, 대형 공지사항 등을 실시간으로 송출하기 위해 만들어졌습니다.

관리자(학생회/운영진)가 스마트폰이나 태블릿으로 컨트롤 패널(어드민)에서 점수를 올리면, 전광판(대형 디스플레이)이나 개인 스마트폰으로 접속한 모든 학생의 화면에 **새로고침 없이** 즉시 반영됩니다.

## 1. 프로젝트 개요

### 프로젝트 정보

- **프로젝트명:** DIMI-score
- **목적:** 체육대회 종목별 실시간 점수 현황, 경기 진행 상태, 타이머 동기화 및 전체 공지사항 관리
- **프로그래머 정보:** 24대 학생회 IT부 차장 WP 제시원

### 주요 기능

- **실시간 점수판:** 청팀/백팀의 실시간 점수 및 경기 현황(진행 전/진행 중/완료) 자동 전환 송출
- **관리자 패널 (Admin):** 버튼 터치 및 직접 입력을 통해 점수, 경기 상태, 타이머, 공지사항을 원격으로 실시간 제어
- **타이머 기능:** 경기 전체 진행 시간(Duration) 및 세트별 카운트다운(Set Duration) 클라이언트 간 완벽 동기화
- **공지사항:** 화면 전체를 덮는 대형 긴급 공지 및 화면 하단에 흐르는 마키(Marquee) 스크롤 공지사항 송출

## 2. 기술 스택

후임 개발자분들이 프로젝트를 빠르게 파악할 수 있도록 각 기술의 도입 이유를 정리했습니다.

- **Frontend:** `React 18`, `TypeScript`, `Vite`
    - 빠른 빌드 속도(Vite)와 런타임 에러 방지(TypeScript)를 위해 채택했습니다.
- **Styling & UI:** `Tailwind CSS`, `Radix UI` (Shadcn UI 기반), `Lucide React` (아이콘)
    - 체육대회 준비 기간이 짧은 만큼, 처음부터 CSS를 짜기보다는 검증된 UI 컴포넌트(Shadcn)를 조립하여 빠르고 예쁘게 개발하기 위해 사용했습니다.
- **State Management:** `Zustand`
    - Redux에 비해 설정(Boilerplate)이 매우 적고 직관적이어서 주니어 개발자도 쉽게 전역 상태를 관리할 수 있습니다.
- **Backend & BaaS:** `Supabase` (PostgreSQL, Auth, Realtime 구독)
    - 별도의 백엔드 서버(Node.js, Spring 등)를 구축하고 소켓 서버를 열 필요 없이, DB 변경 사항을 즉시 클라이언트로 쏴주는 **Realtime 기능**을 무료로 쉽게 쓰기 위해 채택했습니다.

## 4. 주요 폴더 및 파일 구조

어떤 기능을 수정하고 싶을 때 어디를 찾아봐야 하는지 정리한 지도입니다.

- `src/pages/` : 사용자가 접속하는 실제 전체 화면(페이지) 단위 컴포넌트
    - `Index.tsx`: 학생들이 보는 **메인 전광판 화면**입니다. 화면 모드(예: 진행 중, 대기 중, 시간표)에 따라 UI가 전환됩니다.
    - `Admin.tsx`: 운영진이 점수와 타이머를 조작하는 **관리자 패널**입니다.
- `src/components/` : 버튼, 타이머, 오버레이 등 화면을 구성하는 개별 부품들
    - `TimetableView.tsx`, `InProgressView.tsx` 등 특정 뷰를 담당하는 컴포넌트들이 있습니다.
- `src/components/ui/`: Shadcn UI 명령어로 자동 생성된 기본 UI 컴포넌트들 (버튼, 인풋 등)입니다. 가급적 수정하지 않는 것이 좋습니다.
- `src/store/useEventStore.ts`: **이 프로젝트의 심장(핵심 로직)입니다.** 점수 더하기, 상태 바꾸기, Supabase DB 통신, 실시간 구독(Subscribe) 로직이 모두 이 파일 하나에 모여 있습니다.
- `src/integrations/supabase/`: Supabase와 통신할 때 데이터베이스의 테이블 구조(타입)를 미리 정의해둔 파일(`types.ts`)이 있습니다. DB 컬럼이 바뀌면 이 파일도 수정해야 합니다.

## 5. 핵심 아키텍처 및 실시간 동기화 동작 원리

이 코드를 처음 본다면 "점수가 어떻게 실시간으로 모든 폰에서 바뀌지?"가 가장 궁금할 것입니다. 아래 4가지 원리를 꼭 이해하고 코드를 수정하세요.

1. **중앙 상태 관리 (`useEventStore.ts`)**
버튼을 누르면 먼저 Zustand 스토어의 내부 상태를 업데이트하여 화면을 즉시(0.1초 만에) 바꾼 뒤(Optimistic Update), 뒤이어 Supabase DB에 값을 저장하도록 설계되어 있어 체감 속도가 매우 빠릅니다.
2. **실시간 동기화 (Supabase Realtime)**
Supabase의 `dimi-score-sync` 채널을 열어두고(WebSocket), 누군가 DB의 `app_state`나 `events` 테이블을 수정하면 그 변경사항을 즉시 감지해 모든 접속자의 화면 데이터를 덮어씌웁니다.
3. **안전장치 (Polling Fallback) - 절대 지우지 마세요!**
체육대회는 수백 명이 한 번에 모이는 야외 행사이므로 **와이파이나 데이터가 끊기거나 느려질 확률이 높습니다.** 소켓(Realtime) 연결이 끊어질 경우를 대비해, **2초(2000ms) 간격으로 일반적인 API 요청(Polling)을 보내 데이터를 억지로 맞춰오는 백업 로직**이 구현되어 있습니다.
4. **비용 효율적인 타이머 관리 (Realtime Broadcast)**
초 단위로 줄어드는 타이머를 DB에 매초 저장하면 서버 비용(트래픽)이 폭발합니다. 따라서 '시작 버튼을 누른 정확한 타임스탬프(시간)'만 Broadcast 채널로 한 번 전송하고, 각 기기의 자바스크립트가 알아서 현재 시간과 비교하여 타이머를 계산해 보여주도록 최적화되어 있습니다.

## 6. 데이터베이스 스키마 및 초기 데이터 (Supabase)

Supabase 대시보드에 직접 들어가면 확인할 수 있는 핵심 테이블 2개입니다. 새로운 컬럼을 추가하려면 테이블 구조부터 변경해야 합니다. (`supabase/migrations/` 폴더의 SQL 참조)

- **`app_state` 테이블** (전역 설정: ID 1번 줄 하나만 계속 업데이트하며 씁니다)
    - `view_mode`: 현재 전광판이 띄워야 할 모드 (TIMETABLE, IN_PROGRESS 등)
    - `announcement`: 화면 하단에 지나가는 공지 텍스트
    - `announcement_timestamp`: 대형 공지를 띄울 때 사용하는 시간 트리거
    - `bonus_score_a`, `bonus_score_b`: 종목과 관계없이 부여하는 청팀/백팀 기본 보너스 점수
- **`events` 테이블** (경기 종목 리스트)
    - `id`, `name`(종목명), `time`(시작 시간)
    - `status`: 현재 경기 상태 (`UPCOMING`(진행 전), `IN_PROGRESS`(진행 중), `COMPLETED`(완료))
    - `score_a`, `score_b`: 해당 종목에서 청팀/백팀이 얻은 점수

## 7. 관리자 페이지 매뉴얼

실제 체육대회 당일 운영진이 사용할 `Admin.tsx`(`/admin` 경로)의 사용법입니다.

> **상세 매뉴얼 노션 링크:** [운영진 가이드 보러가기](https://www.notion.so/343fe0074be78047844ec6cdf42067fb?pvs=21)
> 
- **로그인 (보안):** 하드코딩을 방지하기 위해 Supabase Auth 기능으로 로그인해야 접속할 수 있습니다.
- **전체 점수판 관리:** 청/백팀 전체 총점을 확인하고, 종목 외의 가산점(태도 점수, 응원 점수 등)을 +/- 버튼이나 수동 입력으로 부여할 수 있습니다.
- **공지사항:** 하단 스크롤 공지를 입력하고 '전송' 하거나, 경기 중단 등 긴급 상황 시 '대형 공지 띄우기'로 화면 전체를 덮을 수 있습니다.
- **종목 관리 (하단):** 각 종목을 찾아 '진행 중' 버튼을 누르면 메인 화면이 해당 경기의 점수판으로 전환됩니다. 여기서 각 팀의 점수를 +1, +2, +3 등 버튼으로 실시간 반영할 수 있으며 타이머 카운트다운을 시작/초기화할 수 있습니다.

## 8. 배포 및 인프라 설정 (Deployment)

본 프로젝트는 **Vercel**을 통해 자동 배포되도록 구성되어 있습니다. (`vercel.json` 포함)

1. Github에 코드를 Push하거나 `main` 브랜치에 병합(Merge)하면 Vercel이 자동으로 변경된 사이트를 배포합니다.
2. **가장 중요한 주의사항:** 로컬 개발 환경에서 `.env` 파일에 넣었던 `VITE_SUPABASE_URL`과 `VITE_SUPABASE_PUBLISHABLE_KEY`를 **반드시 Vercel 대시보드 -> Settings -> Environment Variables 탭에 똑같이 등록해 주어야 합니다.** 등록하지 않으면 배포된 사이트가 DB를 찾지 못해 백지상태가 되거나 에러를 발생시킵니다.

## 9. 자주 발생하는 문제 및 해결 방법

개발 및 리허설 과정에서 당황할 수 있는 대표적인 문제 2가지입니다.

- **Q. 관리자 페이지에서 점수를 올렸는데 메인 화면이 즉시 안 바뀌고 2초 정도 늦게 바뀝니다!**
    - A. 실시간(Realtime) 소켓 통신이 실패하고 Polling(2초마다 백업 요청)으로 동작하고 있다는 뜻입니다. Supabase 대시보드에서 `app_state`와 `events` 테이블 설정에 들어가 **Enable Realtime(Replication) 스위치가 켜져 있는지(ON)** 반드시 확인하세요.
- **Q. 관리자 페이지 접속 이메일과 비밀번호는 어디서 추가/수정하나요?**
    - A. 코드에는 아이디/비밀번호 정보가 없습니다. Supabase 대시보드 좌측 메뉴의 **Authentication -> Users** 탭에서 운영진의 이메일과 임시 비밀번호를 수동으로 생성(Add user)해 주면 됩니다. 브라우저 캐시 문제라면 Application 탭 -> Local Storage를 비우고 강력 새로고침(Ctrl + Shift + R)을 해보세요.

## 10. 추가적으로 구현 및 보안해야 하는 기능

26년도 체육대회는 성황리에 잘 마무리되었지만, 현재 사이트 구조상 몇 가지 보완할 점이 있습니다. 27년 체육대회에서는 다음 기능들을 우선적으로 점검하고 고도화하시기를 권장합니다.

### 1. 등수 기반 점수 계산 기능 추가

- **현황:** 계주와 같이 등수에 따라 차등 점수를 부여하는 경기에서는 현재의 +/- 버튼만으로 빠르게 점수를 합산하기 어렵습니다.
- **개선 제안:** 등수(1등, 2등 등)를 선택하면 자동으로 정해진 배점(예: 300점, 150점)이 계산되는 기능을 구현하세요. 특히 빛 반사가 심한 오후 야외 경기에서는 점수를 눈으로 확인하며 계산하기 어려우므로 자동화가 필수입니다.

### 2. 사운드 이펙트 추가

- **현황:** 26년도 대회 당시 사이트의 시각적인 UI와 득점 이펙트가 좋은 반응을 얻었으나 청각적 요소는 부족했습니다.
- **개선 제안:** 점수가 올라갈 때 울리는 경쾌한 득점 효과음이나 타이머 종료 시 울리는 버저비터 사운드를 추가하여 현장감을 극대화하시길 바랍니다.

### 3. 경기별 맞춤형 타이머 구조

- **현황:** 농구 등 수시로 경기가 중단되는 종목은 심판의 휘슬에 맞춰 타이머를 일시 정지해야 하지만, 현재는 시작 시간의 동기화만 가능합니다.
- **개선 제안:** '일시 정지'와 '재개' 기능을 지원하는 타이머 로직으로 개편하거나, 전체 경기 시간만 자동으로 띄워주는 별도의 타이머 모드를 추가하세요.

### 4. 어드민 페이지 모바일 조작 간편화

- **현황:** 화면이 작은 특정 모바일 기기에서 접속 시 어드민 페이지 하단이 스크롤되지 않거나 잘리는 문제가 발생합니다.
- **개선 제안:** 체육대회 현장에서는 대부분 스마트폰을 통해 점수를 조작합니다. 작은 모바일 기기에서도 모든 버튼이 원활하게 터치되고 스크롤될 수 있도록 반응형 UI를 디버깅 및 수정하세요.

### 5. 어드민 페이지 로그인 보안 강화

- **현황:** 과거 하드코딩된 비밀번호에서 현재는 이메일 로그인으로 변경되었으나, 계정을 하나하나 추가해야 하는 불편함과 권한 관리의 한계가 존재합니다.
- **개선 제안:** 허락된 운영진만 안전하고 편리하게 접근할 수 있도록 보다 체계적이고 강력한 로그인 방식(OAuth 연동, RLS 정책 고도화 등)으로 변경하시길 추천합니다.
